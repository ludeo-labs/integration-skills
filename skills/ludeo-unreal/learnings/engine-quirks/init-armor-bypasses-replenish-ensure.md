# Use `Init<Attr>` (not `SetNumericAttributeBase`) when the attribute is chunk-bookkept and replenish-checked

## Precondition

You're writing Player Flow restore code, and the engine's GAS attribute set has a chunk-based attribute — typically armor — with these properties:

1. The runtime value is constrained to chunk-aligned values (`Armor = ChunkCount × ArmorPerChunk - Trauma` or similar formula).
2. A periodic replenish callback recomputes the value: `NewArmor = GetReplenishedArmor(CurrentArmor)` where `GetReplenishedArmor` rounds up to the next full chunk boundary.
3. The replenish callback ends with an `ensure(GetReplenishedArmor(NewArmor) == NewArmor)` — i.e. "the new value must be replenish-idempotent." Any non-chunk-aligned value violates this.

If your captured armor mid-fight is `2.5 chunks worth` and you restore it via `ASC->SetNumericAttributeBase(GetArmorAttribute(), 2.5_value)`, the GAS change-callback chain reaches the replenish callback synchronously, the ensure fires, and you get spammy log errors at minimum (and potentially behavior breakage if the assertion is upgraded to `check`).

## Reference incident (ActionGame)

`APlayerStateBase`, inside `OnArmorReplenish`:

```cpp
ensureMsgf(AttributeSet->GetReplenishedArmor(AttributeSet->GetArmor()) == NewArmor,
    TEXT("%f != %f, %s"),
    AttributeSet->GetReplenishedArmor(AttributeSet->GetArmor()), NewArmor,
    *AttributeSet->ToString());
```

Restoring captured armor via `SetNumericAttributeBase` triggered this ensure. The attribute set's previous "skip the restore entirely" comment had been live for ~6 weeks before being addressed.

## Fix — use the `Init<Attr>` accessor

UE's `GAMEPLAYATTRIBUTE_VALUE_INITTER(Attr)` macro (used by the game's `GAME_DEFINE_ATTRIBUTE_ACCESSORS` macro chain) generates:

```cpp
FORCEINLINE void Init##Attr(float NewVal)
{
    Attr.SetBaseValue(NewVal);
    Attr.SetCurrentValue(NewVal);
}
```

This mutates the `FGameplayAttributeData` field directly. **It does not go through `UAbilitySystemComponent::SetNumericAttributeBase` and therefore does not fire the GAS attribute-change callback chain.** `OnArmorReplenish` is not invoked synchronously, so the replenish-idempotency ensure can't fire from this call.

```cpp
// Get the attribute set instance from the ASC and call InitArmor directly.
// GetSet<T>() returns const T*; Init* is non-const, so cast.
if (UGamePawnAttributeSet* MutablePawnAS = const_cast<UGamePawnAttributeSet*>(
    ASC->GetSet<UGamePawnAttributeSet>()))
{
    MutablePawnAS->InitArmor(CapturedArmor);
}
```

Required: an `#include` for the attribute-set header (the game's pawn attribute-set header for ActionGame, may differ per game).

## Why this is safe long-term

The next replenish tick (~5-10 s after restore) calls `OnArmorReplenish` itself with `NewArmor = GetReplenishedArmor(CurrentArmor)`. The ensure then evaluates `GetReplenishedArmor(NewArmor) == NewArmor` — trivially true by replenish-idempotency. The chunk bookkeeping (`ArmorChunkCount`) is recomputed from the restored `Armor` value during this tick, so the engine's runtime state converges naturally without our involvement. Worst case: if the captured value lands mid-chunk (e.g., 2.5), it rounds up to the next full chunk (3.0) on the first replenish — which matches engine behavior outside Ludeo, where replenish only rounds up.

Trade-off: brief mid-chunk-to-chunk-boundary snap on first replenish tick after replay starts. Acceptable for any replay where armor state matters; the alternative (skipping the restore entirely) starts every replay at full armor.

## How to apply on other games

1. **Check the GAS attribute-change callback path** for the chunk-bookkept attribute. If it includes a replenish step that ensures idempotency, `SetNumericAttributeBase` is unsafe for arbitrary values.
2. **Look for the `Init<Attr>` accessor** — generated by `GAMEPLAYATTRIBUTE_VALUE_INITTER` (UE built-in) or the engine's wrapper macro. If the macro chain that defines the attribute uses a `GAME_DEFINE_ATTRIBUTE_ACCESSORS`-style macro (or any similar macro that includes the value-initter), the `Init*` method exists and is FORCEINLINE / public.
3. **Use the Init path for restore.** Don't go through `SetNumericAttributeBase` for chunk-bookkept attributes.
4. **Trust the replenish tick** to converge bookkeeping (ChunkCount, etc.) from the restored value within seconds. Don't try to manually compute and set the chunk count too — it's redundant and may be overwritten anyway.
