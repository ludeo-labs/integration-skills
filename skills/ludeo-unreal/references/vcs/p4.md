# VCS Contract — Perforce (p4)

Implementation of the VCS contract for Perforce. Loaded when `integration.json → vcs.type == "p4"`. See `README.md` for the operation list and `detect_vcs`. Each operation below is self-contained — if you know git but not Perforce, the key shift is that a p4 workspace is read-only by default (see the read-only rule below) and you submit changelists to a central server rather than committing locally.

State recorded in `integration.json → vcs.p4`:
```json
"p4": {
  "port": null,                      // P4PORT (often from env / P4CONFIG; null = inherit)
  "client": "<workspace-name>",      // verified, not created
  "stream": "//depot/<game>-ludeo",  // or null if changelist-only
  "depotPath": "//depot/<game>/...",
  "mcp": "perforce",                 // detected Perforce MCP server name, or null (CLI fallback)
  "changelist": null                 // active pending CL number for the work
}
```

## Operation stack (how p4 ops run)

1. **Primary — Perforce MCP server.** If a Perforce MCP is connected (default-recommend the **official Perforce P4 MCP**), call its tools for `edit`/`add`/`shelve`/`submit`/stream ops. This is the idiomatic, reliable path and is cross-agent.
2. **Fallback — raw `p4` CLI.** When no Perforce MCP is connected, run the equivalent `p4` commands directly. Every operation below shows the CLI form.
3. **Avoid `allwrite`.** Do not reconfigure the client to make files writable as a shortcut — Perforce discourages it (loses clobber protection). Open files for edit instead.

Record the detected server (or `null`) in `vcs.p4.mcp` during the Per-Session Flow MCP check.

## ⚠ The read-only rule — `ensure_editable(path)` (applies to EVERY write)

A p4 workspace is **read-only by default**: every tracked file on disk is read-only until opened for edit. The harness Write/Edit tools **fail** on a read-only file. So **before any Write/Edit to a workspace file**, open it:

- **Existing file** → open for edit:
  - MCP: the Perforce `edit` tool on `<path>` (into the work changelist).
  - CLI: `p4 edit -c <CL> "<path>"`
- **New file** → mark for add:
  - MCP: the Perforce `add` tool on `<path>`.
  - CLI: `p4 add -c <CL> "<path>"`

This is not a one-time setup step — it is in force for the **whole** integration, every phase, including `.ludeo/` state files and any `.uasset` the BP Inspector modifies (see Gotchas). A short reminder of this lives in the SKILL.md body so it stays in context even when this file isn't re-read.

## `create_isolation(name)`

First determine whether the depot is **stream-based** or **classic**, then propose the matching isolation and **let the integrator confirm** (D2 — never create or switch streams unprompted):

```bash
p4 -ztag -F %Stream% client -o    # non-empty => the client is bound to a stream
```

- **Stream-based depot** — propose a **task or dev stream** off the current stream and ask:
  > "This workspace is on stream `<current>`. To isolate the Ludeo work I recommend a task stream `//<depot>/<game>-ludeo` branched from `<current>`. Want me to use that name (or one matching your convention)? And should I create + switch to it, or would you prefer to create it yourself per your studio's stream policy? I won't create or switch streams without your go-ahead."

  After confirmation (if asked to proceed):
  ```bash
  p4 stream -t development -P <current-stream> //<depot>/<game>-ludeo   # create
  p4 client -s -S //<depot>/<game>-ludeo                                 # switch client to it
  p4 sync
  ```
  Record `vcs.p4.stream`.

- **Classic (non-stream) depot** — there is no branch to switch to; isolate the work in a **dedicated numbered pending changelist** and ask:
  > "This depot isn't stream-based, so I'll keep all Ludeo work in a dedicated pending changelist (description `Ludeo integration — <game>`) rather than submitting to the mainline until you review. OK?"

  ```bash
  p4 --field "Description=Ludeo integration — <game>" change -o | p4 change -i   # prints "Change N created"
  ```
  Record the CL number in `vcs.p4.changelist` and leave `vcs.p4.stream = null`. All `edit`/`add` use `-c <CL>`.

In both cases, all subsequent `edit`/`add`/`reconcile` operations target the work changelist.

## `acquire_component(name, dest)`

p4 has no submodule concept — fetch from the **release zip**. The plugin's release asset is **self-contained**: it bundles the C SDK already populated at `Source/LudeoSDK/SDK/`, so this is a **single download** that satisfies both components (see `config/sdk-sources.json → ludeoUESDKPlugin.release`).

If `Plugins/LudeoUESDK` is already present (e.g. the studio vendored it into the depot) → confirm. Otherwise:

1. Download the release zip from `config/sdk-sources.json → ludeoUESDKPlugin.release.downloadUrl` (or `gh release download <tag> -R ludeo-labs/unreal-plugin-releases -p '*.zip'`). It is large (~816 MB; ~4 GB extracted).
2. Extract into `Plugins/LudeoUESDK`.
3. Mark the extracted tree for add (so it's tracked):
   - MCP: the Perforce `add` tool on `Plugins/LudeoUESDK/...`.
   - CLI: `p4 add -c <CL> "Plugins/LudeoUESDK/..."`
4. **Validate** the expected artifacts landed — `Source/LudeoSDK/SDK/Bin/Win64/Release/LudeoSDK-Win64-Release.dll` (and `Lib/...`, `Include/...`) should exist. If missing, the zip was wrong or incompletely extracted — stop and tell the human.

Record `sdkSetup.uePlugin.method = "zip"`, `tag`, `downloadUrl`, `path`; `sdkSetup.cSdk.method = "bundled-in-plugin-zip"`. Respect `.p4ignore` — do not add the game's `Binaries/ Intermediate/ Saved/`. (The SDK's own prebuilt `Bin/`/`Lib/` ARE build inputs and must be added.) Prefer the release asset over a GitHub source-archive zip, which would not resolve LFS binaries.

## `reconcile` safety-net (end of every code-producing phase)

The agent can forget an `ensure_editable` on some write, silently leaving a file out of the changelist. At the **end of each code-producing phase** (with the compile/verification gate), reconcile to catch this — but **ask the integrator to confirm first**, because reconcile opens/reverts files:

> "I'm about to run `p4 reconcile` on the integration paths to make sure every change I made is opened in changelist `<CL>` (some writes may not have been checked out explicitly). This will open added/edited/deleted files. Proceed?"

Preview first, then reconcile on confirmation:
```bash
p4 status                                   # preview (read-only): what reconcile would open
p4 reconcile -c <CL> -aed "<project>/Source/..." "<project>/Plugins/Ludeo..." "<project>/.ludeo/..."
```
Report what it opened. Scope the paths to the integration's directories — never `reconcile` the whole workspace.

## `open_review(summary)`

Only when the human requests it, after the compile-fix gate passes. There are no pull requests:

- **Shelve** the work changelist so reviewers can see it without a submit:
  - MCP: the Perforce `shelve` tool on the CL.
  - CLI: `p4 shelve -c <CL>`
- **Create a Swarm review** from the shelved changelist (the official Perforce P4 MCP can drive Swarm directly; otherwise hand the integrator the CL number to open the review).

Iterate on review feedback; capture corrections as learnings. Submit (`p4 submit -c <CL>`) only when the human approves.

## `guard_destructive`

Never run an irreversible p4 command based on a single failed check — confirm state with at least two independent checks (absolute paths) first. Treat these as destructive and never run them speculatively:

`p4 revert` (discards your checkout) · `p4 sync -f` (force-overwrites local files) · `p4 client -d` · `p4 obliterate` (admin).

A Perforce MCP with safety defaults helps here — keep `P4_READONLY_MODE`/`P4_DISABLE_DELETE` semantics in mind: write/delete operations should be deliberate and confirmed, not speculative.

## Perforce-specific gotchas

- **BP Inspector writes `.uasset` files.** `RunBPInspector.bat set-savegame` modifies Blueprint `.uasset`s, which are read-only in a p4 workspace. **`ensure_editable` the target `.uasset`(s) first**, or the tool fails. Re-run inspect afterward to confirm.
- **`.p4ignore` already governs ignores** (`Binaries/ Intermediate/ Saved/ *.sln …`). Don't add generated/build output to the depot.
- **You're online.** Most ops talk to the server; there's no local commit to amend. A `submit` is immediately visible to everyone — keep the work in the pending CL / stream until review.
- **`-noP4` in the build** is fine and already set in `BuildAndPackage.bat` — it only tells UAT not to do its own P4 operations during cook; it doesn't affect our checkout/submit flow.
